## 여는 글

지금까지 프로젝트를 진행해오면서 React 프로젝트에서 흔히 볼 수 있는 `components`, `hooks`, `pages`, `layouts` 등으로 구성된 폴더 구조를 사용해 왔는데,
`components` 폴더 내부가 깊어지거나 너무 많은 컴포넌트들이 생기거나 하는 등의 문제가 발생하는 이유가 폴더를 구분하는 명확한 기준이 없어서 라는 생각이 들었어요.

그래서 최근 시작한 프로젝트에 도입하기 위한 새로운 아키텍처나 디자인 패턴을 찾아보다가 주변에서 들어 보았지만 자세히는 몰랐었던 FSD 라는 아키텍처에 대해 알아보게 되었어요.  
새로운 프로젝트에 FSD를 도입하기 위해 찾아보며 알게 됐던 내용들을 정리해보려고 해요.

[공식 문서](https://feature-sliced.design/kr/)에서는 FSD가 프로젝트나 팀의 규모에 상관없이 사용할 수 있다고 설명하면서,  
점진적으로 적용할 수 있으며 이에 더해 [마이그레이션 가이드](https://feature-sliced.design/kr/docs/guides/migration/from-custom)까지 제공하고 있어 확장성에 열려있는 구조라는 생각이 들었어요.

그럼 이제 FSD에 대해서 조금씩 알아볼게요.

FSD를 찾아보면 제일 많이 볼 수 있는 이미지로, FSD는 폴더 구조를 `레이어`, `슬라이스`, `세그먼트`로 이루어진 **3-Depth**를 기본으로 제안하고 있어요.

각 항목에 대한 의미를 알아볼게요.

### 레이어

1. **App** : `Router`, `Provider` 컴포넌트 등 전역 설정과 같은 요소를 말해요.
2. **Pages** : `Router`에 따라서 분리된 주소별 페이지들을 말해요.
3. **Widgets** : 여러 기능과 UI를 조립한 것으로 사이드바, 헤더 같은 인터페이스를 말해요.
4. **Features** : 비즈니스 로직과 같이 특정 행동을 말해요.
5. **Entities** : 데이터 그 자체를 의미하고 컴포넌트에서 사용할 데이터를 말해요.
6. **Shared** : `hook`, `type`, `util` 등 재사용 가능하거나 특정 비즈니스 로직이 아닌 공통의 요소를 말해요.

- `Processes`는 더 이상 사용되지 않아요.
- `App`과 `Shared`는 하위 슬라이스를 가지지 않고 세그먼트로 이루어져요.

레이어의 각 숫자는 순서가 아니라 위계를 의미하고, 상위 레이어는 하위 레이어에 의존할 수 있지만 그 반대는 성립하지 않는 단방향 의존성을 추구해요.  
그래서 각 요소를 변경할 때 생기는 사이드 이펙트를 어느정도 예상가능한 범위로 한정할 수 있게 해준다는 생각이 들었어요.

처음 FSD를 적용하는 경우 `App`, `Page`, `Entities`, `Shared` 이 네가지 폴더로 시작하고  
이후 프로젝트 규모가 커지는 경우 레이어를 추가해서 각 요소들을 레이어에 맞게 구분해 나가면 심리적 허들을 줄일 수 있을것 같아요.

### 슬라이스

슬라이스는 도메인별로 코드를 분할하는 것으로 같은 도메인에서 사용되는 모듈끼리 응집시키는 단위로,  
예를 들어 블로그 프로젝트라면 다음과 같이 구성할 수 있어요.

```plain
entities    ## 레이어
 ┗ user     ## 사용자와 관련된 세그먼트들이 포함된 슬라이스
 ┗ post     ## 블로그 게시글과 관련된 세그먼트들이 포함된 슬라이스
 ┗ comment  ## 댓글와 관련된 세그먼트들이 포함된 슬라이스
```

### 세그먼트

세그먼트는 목적에 따라 그룹을 만들어 사용하는데, 몇가지 관례적인 이름은 다음과 같아요.

- `ui` : UI와 관련된 모든 것들(컴포넌트)이 들어가요.
- `api` : 백엔드와 상호작용할 로직이 들어가요.
- `model` : 스키마, 인터페이스, 스토어와 같은 비즈니스 로직이 들어가요.
- `lib` : 다른 모듈이 필요로 하는 라이브러리 코드가 들어가요.
- `config` : 설정 파일과 상수가 들어가요.

## 적용

FSD에서 사용하는 구조를 알아보았으니, 이제 기준을 정해볼건데, 이 기준은 팀 내부에서 각자의 기준을 정해 사용하면 좋을것 같아요.  
저는 하위 슬라이스가 없고 공통의 성격을 띄는 `App`, `Shared`부터 시작해서 하위 구성, 상위 조합 순서로 정리해봤어요.

### src/app

> `Provider`, `Theme` 등의 전역 컴포넌트, 설정을 포함해요.

- **/ui** : 전역 레이아웃 또는 테마 (`AppLayout`, `ThemeProvider`)
- **/api** : 전역 API 클라이언트 설정 (`axios` 인스턴스, `API` 인터셉터)
- **/model** : 전역 상태관리, 타입 및 공통 인터페이스 (`zustand` 스토어, 인증, 권한 관리)
- **/lib** : 프로젝트 전체에서 사용되는 유틸리티 함수 (로깅, 에러 핸들링)
- **/config** : 환경변수, 상수 등의 전역 설정 (`API URL`, `FEATURE_NAME`)

### src/shared

> 여러 프로젝트에서도 재사용 가능한 요소들을 포함해요.

- **/ui** : 재사용하는한 UI 컴포넌트
- **/api** : 공통 API 설정 (`Axios` 인스턴스)
- **/model** :프로젝트에서 공통으로 사용하는 훅
- **/lib** : 공통으로 사용하는 유틸리티 함수
- **/config** : 색상, 반응형 브레이크 포인트 등 전역 상수

### src/entities

> 도메인 별 데이터와 관련된 요소를 포함해요.

- **/ui** : 데이터를 표현하는 컴포넌트
- **/api** : 데이터를 페칭하는 API
- **/model** : 데이터의 타입과 관련된 비즈니스 로직
- **/lib** : 데이터를 위한 유틸리티 함수
- **/config** : 데이터의 상수, 설정 값

### src/features

> 어떤 행동과 사용자 상호작용에 관련된 기능을 포함해요.

- **/ui** : 어떤 행동을 구현하는 컴포넌트
- **/api** : 행동을 위해 호출하는 API
- **/model** : 행동과 관련된 상태 관리, 비즈니스 로직, 타입 정의
- **/lib** : 기능 내에서만 사용되는 유틸리티 함수
- **/config** : 기능에 필요한 상수, 설정 값

### src/widgets

> `features`를 조합한 독립적인 컴포넌트를 포함해요.

- **/ui** : UI들의 조합 (`Header`, `Footer`, `Sidebar`)
- **/api** : 위젯에서 사용하는 데이터 페칭 API
- **/model** : 위젯의 상태 관리, 구조, 타입
- **/lib** : 위젯에서만 사용되는 유틸리티 함수
- **/config** : 위젯에서 사용되는 상수, 설정 값

### src/page

> `HomePage`와 같은 페이지를 포함해요.

- **/ui** : 페이지 레이아웃
- **/api** : 페이지에서 호출하는 API
- **/model** : 페이지 상태 관리, 구조, 타입
- **/lib** : 페이지 내에서만 사용되는 유틸리티 함수
- **/config** : 페이지의 설정 값

## 닫는 글

FDS에 관련된 여러 글들을 살펴보면서 약간의 러닝커브가 있다는 이야기를 보았는데, 저도 실제로 비슷한 느낌을 받았어요.  
먼저 팀원들이 FSD의 각 레이어, 슬라이스, 세그먼트의 구분에 대한 학습이 선행되어야 하고 팀 내부적으로 설정한 기준에 대해 명확한 이해가 있어야 하니까요.

그럼에도 도입에 대해 긍정적인 생각이 들었던 이유는 폴더를 구분하는 기준을 UI에서, 데이터와 기능 위주로 바라본다는 점이 관심사 분리의 측면에서  
기존의 구조에서 느꼈던 불편함들을 어느정도 개선해줄 수 있겠다는 생각이 들었기 때문이에요.

그래서 새로운 프로젝트에 도입해 어느정도 학습해보고, 기존에 작업했던 프로젝트를 마이그레이션 하는 경험을 통해 여러 엣지 케이스를 경험하면 좋을것 같다는 생각이 들었어요.
