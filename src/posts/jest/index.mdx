테스트 코드는 내가 작성한 코드를 검증하는데에 중요한 역할을 하지만, 개발 일정 증가 등의 부담 때문에 우선순위를 낮게 볼 때가 많아요. 하지만 TDD 와 같은 개발 방법론을 선호하는 개발자가 있다는건 테스트의 중요성을 반증한다고 볼 수 있어요.

프론트엔드 개발에서 사용할 수 있는 여러 테스팅 라이브러리 중에서도 가장 널리 사용되는 것은 다음과 같아요:

- **Jest**: Facebook에서 개발한 JavaScript 테스팅 프레임워크에요.
- **Vitest**: Jest와 호환되는 Vite 기반의 테스팅 프레임워크에요.

요즘에 많이 사용하는 추세인 **Vitest**도 **Jest** 문법을 기준으로 하기 때문에, **Jest**에 대해 알아보려고 해요.  
저는 테스트 코드를 작성하는 법에 대해 공부하면서 다음과 같은 목적을 달성해보려고 했어요.

> 1. 테스트 코드를 틈틈이 작성할 수 있도록 친숙해지자.
> 2. 테스트를 위한 테스트가 아니라 의미있는 테스트 코드를 작성해보자.

이 글은 [공식문서](https://jestjs.io/)와 [제로초님의 Jest 강의](https://inf.run/3Veyh)를 바탕으로 정리했어요.

## 시작하기

```bash
npm i -D ts-jest @types/jest
npx ts-jest config:init
```

npm으로 jest 타입스크립트 패키지를 설치하고 초기 설정을 해줬어요.

Jest를 설치하게 되면 `test()`라는 글로벌 함수를 사용할 수 있게 돼요.  
`test()` 함수는 테스트 설명과 실제 테스트 코드 두 개의 인자를 받고, 다음과 같이 작성할 수 있어요.

```tsx
test("이 함수는 이런 동작을 해요.", () => {
  // 테스트 코드
});
```

두 인자 x, y를 더해서 리턴해주는 `sum` 함수를 작성해볼게요.

```tsx
export default function sum(x: number, y: number) {
  return x + y;
}
```

이 함수에 대한 테스트 코드는 다음과 같이 작성해볼 수 있어요.

```tsx
test("sum 함수는 인자로 받은 두 숫자의 합을 리턴해요.", () => {
  expect(sum(2, 3)).toBe(5);
});
```

여기서 `toBe()` 함수는 `expect()` 함수에 전달된 값(여기서는 `sum(2, 3)`의 결과)이 어떤 값이 되어야 하는지에 대한 기대값을 설정해요.

이 테스트에서는 `sum(2, 3)`의 결과가 `5`가 되기를 기대하고 있고 테스트 결과는 `true`가 될거에요.

반대로 테스트하려는 값이 어떤 값이 되지 않는지 확인하는 `not.toBe()` 와 같은 테스트 메소드도 있어요.
